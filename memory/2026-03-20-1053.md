## 0. 全体の使い方
流れはこれで固定すると強い。

1. **variant.ai**
   - 現在デザインを読み取らせる
   - デザインシステム化
   - 複数スタイル・複数レイアウト案を出させる
   - 最終的に**他agentへ渡せる完全prompt**を吐かせる

2. **Replit Agent**
   - variant.ai が吐いた prompt をそのまま投入
   - まず1ページを高再現で復元
   - その後、同じ design system で他ページへ展開
   - 仮データ込みで動くUIまで作る

3. **Claude Code**
   - Replit 生成物をローカルで整理
   - コンポーネント分割、状態管理、型、安全性、保守性を改善
   - 本番実装向けにリファクタ

# 1. variant.ai 用テンプレ
目的：  
**「いま見えているデザインを、再利用可能な設計言語と prompt に圧縮する」**

### variant.ai に入れるテンプレ
```text
You are a design analysis and design-system extraction agent.

Your task:
1. Analyze the current UI/design in detail.
2. Infer the visual language, layout logic, interaction patterns, and design principles behind it.
3. Generate multiple expandable style directions and layout variations that preserve the product intent.
4. Convert the result into a reusable prompt package that can be given to other AI coding/design agents.

Output in the following structure:

## A. Design Summary
- Product vibe / emotional tone
- Target user impression
- Brand/personality keywords
- Core visual principles
- UX priorities

## B. Visual DNA
Describe in detail:
- Color system
- Typography system
- Spacing rhythm
- Grid/layout structure
- Border radius
- Shadows / depth
- Icon style
- Illustration style
- Motion style
- Density level
- Contrast philosophy

## C. Core UI Patterns
List recurring interface patterns:
- Navigation
- Cards
- Tables/lists
- Filters/search
- Forms
- Buttons
- Modals/drawers
- Empty states
- Notifications
- Data visualization blocks

## D. Layout Logic
Explain:
- How information hierarchy is constructed
- How sections are grouped
- How responsive behavior should work
- Which areas are fixed vs flexible
- How users scan the page

## E. Expandable Variants
Create 3-5 design variants derived from the original:
For each variant include:
- Variant name
- Style direction
- What changes visually
- What stays invariant
- Best use cases

Also create 3 layout variations:
- Compact productivity layout
- Balanced default layout
- Editorial / spacious layout

## F. Design Tokens
Output a token-style spec:
- Colors
- Typography
- Spacing
- Radius
- Shadow
- Border
- Motion
- Component states

## G. Rebuild Prompt for Other Agents
Write a complete, high-quality prompt that another AI agent (e.g. Replit Agent) can use to recreate this design faithfully.

The rebuild prompt must include:
- Product goal
- Visual style rules
- Layout rules
- Component rules
- Responsive behavior
- Interaction guidance
- Accessibility expectations
- What to avoid
- Expected output quality bar

## H. Multi-Page Extension Prompt
Write a second prompt for extending this design system into additional pages/features while preserving consistency.

## I. Implementation Handoff Notes
Provide practical implementation notes for frontend agents:
- Suggested component architecture
- Suggested design-token structure
- Risk areas where generated code may drift from the design
- What should be manually reviewed

Be highly concrete.
Do not stay abstract.
Write so another agent can reconstruct the design with high fidelity.
```

# 3. Replit Agent 用テンプレ
目的：  
**variant.ai の設計書を元に、高再現でUIとページ群を実装する**

### Replit 1発目
```text
You are rebuilding a product UI from a design handoff package.

Goal:
Recreate the design faithfully and build a clean, working frontend based on the provided style/system instructions.

Requirements:
1. First prioritize visual fidelity and consistency with the provided design package.
2. Create a clean project structure with reusable components.
3. Implement design tokens for colors, typography, spacing, radius, shadows, and states.
4. Build the page so that it looks polished, production-grade, and consistent.
5. Use responsive design for mobile, tablet, and desktop.
6. Include realistic placeholder content where needed.
7. Keep accessibility in mind:
   - semantic HTML
   - keyboard navigability
   - visible focus states
   - reasonable contrast
8. Do not invent a different design language.
9. Do not simplify the interface into a generic AI-generated dashboard.
10. Match hierarchy, density, spacing, and visual tone as closely as possible.

Implementation expectations:
- Reusable components
- Clean styling system
- Consistent spacing and type scale
- Modular file organization
- Easy to extend into additional pages later

Process:
- First reconstruct the main page/view faithfully
- Then extract reusable UI patterns
- Then prepare the structure for extending more pages

Design handoff package:
[PASTE VARIANT OUTPUT HERE]
```

```text
Using the existing implemented design system and components, now extend the product into additional pages/features.

Rules:
1. Preserve the exact visual language already established.
2. Reuse existing components whenever possible.
3. Do not create unnecessary one-off styles.
4. Keep spacing, typography, radius, shadows, and interaction patterns consistent.
5. Any new UI must feel like it belongs to the same product.

Please create these pages/features:
- [PAGE 1]
- [PAGE 2]
- [PAGE 3]

For each page:
- Maintain consistent navigation and layout logic
- Use the same design tokens
- Respect the same UX density and tone
- Include realistic states:
  - loading
  - empty
  - error
  - populated

Also:
- Refactor shared UI into reusable components where needed
- Keep the project organized and scalable
- Avoid visual drift
```

# 6. Claude Code 用テンプレ
目的：  
**見た目を壊さず、中身を本番運用向けに整理する**

```text
Refactor this codebase for production readiness without changing the established visual design.

Primary constraint:
Do NOT materially change the UI appearance, layout hierarchy, spacing rhythm, or design language unless necessary to fix a bug or inconsistency.

Goals:
1. Preserve the current UX and visual fidelity.
2. Improve code quality, maintainability, and architecture.
3. Extract reusable components where appropriate.
4. Normalize design tokens and shared styles.
5. Improve type safety and state handling.
6. Reduce duplication.
7. Make the project easier to scale with more pages/features.

Please review and improve:
- component boundaries
- folder structure
- repeated styles
- design token organization
- props/interfaces/types
- state management
- data flow
- loading/error/empty states
- accessibility
- responsiveness
- performance footguns

Implementation preferences:
- Keep components small and understandable
- Prefer clear naming
- Avoid overengineering
- Avoid premature abstraction
- Extract only genuinely reusable patterns
- Preserve developer ergonomics

Output:
1. Summary of architectural issues found
2. Refactor plan
3. Implemented refactor
4. Notes on any UI risks or places where manual review is recommended
```

# 9. かなり大事な運用ルール
クロウ視点だと、この方式はかなりいいけど、以下を固定するともっと安定する。

### ルールA: まず「見た目」と「構造」を分離
variant.ai / Replit に最初からこれを意識させる。

- **見た目**
  - colors
  - typography
  - spacing
  - component styling
- **構造**
  - routing
  - data model
  - state management
  - API boundary

見た目の再現と、内部設計の健全性は別問題。

### ルールB: Replit は“増築担当”
- 速く作る
- 雰囲気を揃える
- 画面を増やす
- 仮データで動かす

### ルールC: Claude Code は“整備担当”
- コンポーネント整理
- 型付け
- 状態管理
- 重複排除
- 本番寄せ

### ルールD: 完璧な初回生成を狙わない
最初から100点狙いじゃなくて、
- 1回目: 雰囲気固定
- 2回目: ページ展開
- 3回目: 構造整理
で十分強い。
