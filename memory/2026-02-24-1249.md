## 1) 全体像（アーキテクチャの骨格）
- **Gateway（中核）**  
  長寿命プロセスとして常駐し、チャネル接続・セッション管理・ルーティング・ツール実行の司令塔になる。
- **Messaging surfaces（入口）**  
  WhatsApp / Telegram / Discord / Signal / iMessage など、複数チャネルを同時接続可能。
- **Control-plane clients（運用面）**  
  CLI・Web UI・macOS app が Gateway に WebSocket で接続して監視/操作。
- **Nodes（外部実行面）**  
  iOS/Android/macOS/headlessノードが `role: node` で同じWSへ接続し、camera/canvas/location等を提供。

---

## 2) どう動くか（処理フロー）
1. チャットからメッセージ受信  
2. Gatewayが**バインディング規則**で対象エージェントを決定  
3. セッションキー解決（DM/グループ/チャネル単位）  
4. Agent Loop実行（文脈組立 → モデル推論 → ツール呼び出し → ストリーム返信）  
5. 結果を返信し、セッション状態をGateway側に永続化

ポイントは、**Gatewayが唯一の真実源（source of truth）**で、UI側はそれを読むだけという設計。

---

## 3) セッションとマルチエージェント
- **単一エージェント**でも使えるし、**複数エージェント分離**も可能。  
- 各エージェントは基本的に  
  - 専用workspace  
  - 専用agentDir（認証プロファイル等）  
  - 専用sessionsストア  
  を持つ。
- DMセッションは `session.dmScope` で粒度を制御（`main` / `per-peer` / `per-channel-peer` など）。  
  複数人DM運用では `per-channel-peer` などの分離設定が重要。

---

## 4) 機能（何ができるか）
- **マルチチャネル統合**（複数サービスを1 Gatewayで運用）
- **マルチエージェントルーティング**（用途/送信元/アカウント別に分岐）
- **ツール実行基盤**（exec/process, browser, web_fetch, nodes, cronなど）
- **メディア対応**（画像/音声/文書）
- **Web Control UI / CLI / macOS連携**
- **モバイルノード連携**（カメラ・Canvas・位置情報など）
- **自動化**（cron, hooks, heartbeat）

---

## 5) 運用・セキュリティの要点
- 推奨は**1ホスト1Gateway**（特にWhatsAppセッションの一貫管理）。  
- 非ループバック公開時はGateway認証トークン必須。  
- 設定は `~/.openclaw/openclaw.json`（JSON5）で、厳格バリデーション。  
- Tool allow/deny・profileで権限を絞るのが基本。  
- 監査は `openclaw security audit` が起点。
