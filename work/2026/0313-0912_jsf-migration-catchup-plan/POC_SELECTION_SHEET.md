# PoC対象選定シート

## 目的
本番移行前に、代表画面や代表機能を使って移行難易度・非互換・実行基盤適合性を確認する。

## 2. 候補ごとの評価観点
各候補を 1（低）〜5（高）で採点する。

### 業務観点
- [ ] 業務影響が大きい
- [ ] ユーザー利用頻度が高い
- [ ] 障害時の影響が大きい

### 技術観点
- [ ] JSP / Facelets の変換難度が高い
- [ ] `@ManagedBean` → CDI 置換が多い
- [ ] バリデーション / コンバータが複雑
- [ ] Ajax / 部分更新が多い
- [ ] UIライブラリ依存が強い
- [ ] 認証 / セッション依存がある
- [ ] ファイル入出力がある

---

## 3. PoCで確認すべき項目
### 共通確認
- [ ] Java 21でビルドできる
- [ ] Jakarta Faces 4 で起動できる
- [ ] `javax.*` → `jakarta.*` 置換で通る
- [ ] 対象サーバで動作する
- [ ] 例外ログ / 起動ログが許容範囲

### 画面確認
- [ ] 初期表示
- [ ] 入力 / バリデーション
- [ ] エラーメッセージ表示
- [ ] 画面遷移
- [ ] Ajax部分更新
- [ ] セッション継続
- [ ] 認可制御

### 開発生産性確認
- [ ] ビルド設定が整理できる
- [ ] テスト方針が立つ
- [ ] 修正量を概算できる
- [ ] 横展開の難度が想定できる

---

## 4. 推奨PoCパターン

### PoC-A: 最小構成検証
**目的**: 実行基盤と基本動作確認  
**対象**: 単純入力画面  
**確認**: 起動 / CDI / バリデーション / 画面遷移

### PoC-B: 代表画面移植
**目的**: 実案件の難所確認  
**対象**: Ajaxや一覧検索を含む実画面  
**確認**: 依存ライブラリ / 部分更新 / 画面状態保持

### PoC-C: 高リスク領域検証
**目的**: ブロッカー把握  
**対象**: JSP画面、旧ライブラリ依存画面、ファイルアップロード、認証連携画面など  
**確認**: 単純移行可否 / 再設計必要性

---

## 5. 推奨選定ルール
PoC対象は最低3本。

1. **低難度1本**: 最小構成確認
2. **中難度1本**: 実運用に近い主要画面
3. **高難度1本**: 最も壊れそうな画面

これで、
- 動くか
- どこで詰まるか
- 横展開できるか
を判断しやすい。

## 7. 判定
### Go
- 代表画面で主要機能が再現可能
- ライブラリ・基盤の目途が立つ
- 工数見積の精度が上がる

### Conditional Go
- 技術的には可能だが、特定領域に再設計が必要
- 追加PoCや段階移行が必要

### Re-plan
- JSP全面刷新やライブラリ総置換が前提
- 単純移行想定を見直す必要あり
